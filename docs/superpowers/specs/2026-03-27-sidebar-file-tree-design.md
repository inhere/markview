# Sidebar File Tree Design

## Goal

在左侧固定侧栏中同时展示目录树和当前文档 TOC。目录树用于文档间导航，TOC 用于文档内导航，布局采用上下双分区，目录树与 TOC 的高度比例固定为 1:2。

## Confirmed UX Decisions

- 侧栏分为上下两块：
  - 上半部分为目录树，约占 33%
  - 下半部分为当前文档 TOC，约占 67%
- 两块区域各自独立滚动，互不影响。
- 目录树中：
  - 目录名点击后优先打开该目录下的 `index.md`
  - 目录右侧单独提供展开/收起箭头
  - 当前文档所在路径自动展开
  - 当前文档或当前目录入口高亮
- 没有 `index.md` 的目录不可导航，但仍可展开查看下级内容。
- TOC 保持现有滚动高亮与锚点跳转行为。

## Data Model

服务端在渲染页面时同步生成目录树数据，并直接注入模板，避免前端再发起额外请求。

目录树节点包含：

- `name`：显示名称
- `href`：点击后的页面路径
- `matchPath`：用于判定当前高亮的相对 Markdown 路径
- `kind`：`directory` 或 `file`
- `navigable`：是否可点击跳转
- `children`：子节点

## Tree Building Rules

- 只展示目录和 Markdown 文件，忽略其他静态资源。
- 跳过 `.git` 和 `node_modules`。
- 目录节点仅在其自身存在 `index.md` 或其后代包含 Markdown 文件时保留。
- 若目录存在 `index.md`，目录名负责导航到该目录入口，同时不在子节点中重复展示该 `index.md` 文件。
- 节点顺序为：目录优先，再按名称排序；文件随后按名称排序。

## UI Structure

侧栏增加两个 section：

1. `Files`
   - section 标题
   - 树形列表容器
   - 目录项左侧显示文件夹图标，右侧显示展开按钮
2. `On This Page`
   - section 标题
   - 沿用当前 TOC 列表

## Implementation Scope

- 后端：
  - 生成目录树结构
  - 将目录树 JSON 与当前文件相对路径写入模板数据
- 前端：
  - 渲染目录树
  - 处理展开/收起
  - 自动展开当前路径祖先节点
  - 高亮当前节点
- 样式：
  - 完成上下分区布局
  - 增加目录树视觉样式
  - 保持现有 TOC 与内容区行为不变

## Risks

- 目录较深时，树节点缩进和截断需要控制，否则可读性会下降。
- 当前目录入口使用目录名跳转 `index.md`，必须避免与展开点击区域冲突。
- 目录树构建逻辑必须与现有请求路径解析保持一致，避免出现能显示但不能访问的节点。
